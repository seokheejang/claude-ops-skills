# ralph — Judgement Policy

FAIL/NOTE 판정 기준 및 협력적 행동 규칙. 매 라운드 적용.

## FAIL vs NOTE 판정

핵심 질문: **"이거 안 고치면 실제로 터지나?"**

### [FAIL] — 즉시 수정

터지는 것 / 동작 자체에 영향:
- 의존성/순서 버그
- 문법 오류 (parse error, type error)
- 존재하지 않는 함수/메서드 호출
- 치환 안 되는 변수 (placeholder가 그대로 남음)
- 보안 취약점 (injection, hardcoded credentials)
- 빈 참조 (404 링크, 미정의 import)

### [NOTE] — 수정하지 않음, 리포트에만 표기

터지지 않는 것 / 스타일 개선:
- 변수명 리네이밍
- 로그 메시지 개선
- "혹시 모르니까" 방어 코드
- 스타일/포맷 통일 (lint warning 수준)
- 주석 추가 제안

→ NOTE는 보고만 하고 자동 수정하지 않는다. 사용자가 원하면 별도 요청.

## 사용자 개입이 필요한 케이스

### 판단이 필요한 이슈

설계 결정/트레이드오프가 얽힌 이슈는 자동 수정 금지. 선택지 제시:

```
[QUESTION] <이슈 설명>
  옵션 A: ...
  옵션 B: ...
  추천: A (이유: ...)
```

### 불확실한 이슈

이게 실제 버그인지 의도된 동작인지 모호하면:

```
[QUESTION] <코드 스니펫>
  의도하신 게 X인가요, Y인가요?
```

## 루프 종료 조건

| 상황 | 처리 |
|------|------|
| 전체 [PASS] | 즉시 완료 (남은 라운드 스킵) |
| 2회 연속 동일 [FAIL] 반복 | 사용자에게 판단 요청 (자동 수정 무한 루프 방지) |
| max-iterations 도달 | 미해결 이슈 목록 정리 후 종료 |
| 사용자 중단 요청 | 즉시 종료, 진행 상태 보고 |

## 협력 원칙

- Ralph는 **일방적으로 수정하지 않는다** — 사용자와 함께 찾는다
- 매 라운드 결과를 공유하며 사용자가 중간에 중단할 수 있음을 안내
- "이게 더 좋다"라고 판단되어도 사용자 의도와 다르면 [NOTE]로만 보고
- 같은 [FAIL]을 두 번 반복 발견하면 자기 판단 의심 → 사용자에게 확인
